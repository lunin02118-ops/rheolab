# Спецификация веб-сайта RheoLab Enterprise

## 1. Обзор проекта

Сайт **RheoLab Enterprise** является публичной витриной продукта и короткой справкой по программе. Его задача не в том, чтобы показывать длинный маркетинговый лэндинг, а в том, чтобы быстро объяснить суть продукта и дать человеку понятный следующий шаг.

Главная мысль сайта:

*   RheoLab хранит результаты экспериментов в одной библиотеке;
*   помогает работать с файлами от разных приборов;
*   позволяет повторно открывать, сравнивать и выгружать результаты в отчёт;
*   даёт короткий путь к скачиванию программы и контакту.

### 1.1. Цели сайта

*   **Понять продукт за минуту:** объяснить реальную пользу без технарщины и перегруза.
*   **Провести к действию:** скачать установщик или связаться с командой.
*   **Поддержать первый запуск:** дать короткую справку и базовые ответы после скачивания.
*   **Согласовать ожидания:** описывать продукт так, как он реально устроен в кодовой базе.

### 1.2. Целевая аудитория

*   лаборанты;
*   полевые и технические специалисты;
*   исследователи;
*   руководители лабораторий.

## 2. Технологический стек

### 2.1. Основной стек

*   **Framework:** Astro 5
*   **Язык:** TypeScript / JavaScript
*   **Рендеринг:** статическая генерация

### 2.2. Контент и стили

*   **CSS:** vanilla CSS с переменными и локальными стилями компонентов
*   **Документация:** Markdown через Astro pages
*   **Формулы в справке:** `remark-math` + `rehype-katex`

### 2.3. Сборка

*   **Скрипты:** `npm run dev`, `npm run build`, `npm run preview`
*   **Выход сборки:** `website/dist/`

## 3. Структура сайта

### 3.1. Основные маршруты

1.  `/` — главная страница продукта
2.  `/docs` — короткая справка по программе
3.  `/privacy` — политика конфиденциальности
4.  `/terms` — условия использования
5.  `/download/latest` — прямой маршрут на актуальный установщик

### 3.2. Главная страница

Главная собрана как короткий маршрут:

1.  **Hero**
    *   короткое позиционирование;
    *   большой график в стиле интерфейса программы;
    *   CTA на скачивание и контакт.
2.  **Capabilities**
    *   какие проблемы решает программа;
    *   что она делает в ежедневной работе;
    *   кому она полезна.
3.  **Download**
    *   прямой маршрут к скачиванию;
    *   мягкий переход в справку и поддержку.
4.  **Contact**
    *   контактные адреса;
    *   форма заявки;
    *   подсказка, что написать для быстрого ответа.
5.  **Footer**
    *   короткие ссылки по сайту, справке и юридическим страницам.

### 3.3. Справка

Справка — это не отдельный тяжёлый портал документации, а короткий support-слой после главной страницы. Она должна:

*   быстро отвечать на практический вопрос;
*   не заводить пользователя в тупик;
*   давать явный путь обратно на главную, к скачиванию или к контакту.

## 4. Визуальная система

### 4.1. Общий характер

*   тёмная профессиональная база;
*   спокойные панели и читаемые контрасты;
*   акцент на живом графике и интерфейсной достоверности;
*   минимум декоративного шума.

### 4.2. Анимированный фон

Ключевая визуальная особенность сайта — фоновый график по секциям.

Требования:

*   фон строится из линий, похожих на реальный экран программы;
*   используются три цветовых канала:
    *   синий — вязкость;
    *   фиолетовый — скорость сдвига;
    *   оранжевый — температура;
*   линии мягко движутся слева направо;
*   каждая секция имеет свой вариант фона;
*   анимация остаётся вторичной и не мешает чтению;
*   для `prefers-reduced-motion` движение отключается или упрощается.

### 4.3. Типографика

*   основной текст: `IBM Plex Sans`
*   display-акценты: `Manrope`
*   моноширинные подписи: `IBM Plex Mono`

## 5. Смысловая модель сайта

Сайт должен описывать продукт как:

*   библиотеку результатов экспериментов;
*   рабочее место для хранения, поиска и сравнения;
*   инструмент, который сохраняет график, данные и отчёт рядом;
*   локальную программу для Windows, а не облачный сервис.

Сайт не должен сводить продукт к:

*   одному графику;
*   одной функции импорта;
*   перегруженной отраслевой витрине;
*   англоязычным маркетинговым терминам без необходимости.

## 6. Контентные правила

### 6.1. Что обязательно подчёркивать

*   одна библиотека результатов;
*   работа с файлами от разных приборов;
*   повторное открытие и сравнение серий;
*   подготовка отчётов;
*   локальное хранение данных;
*   короткий путь к скачиванию и контакту.

### 6.2. Чего избегать

*   внутренних ремарок для команды;
*   машинных формулировок;
*   необязательных англицизмов;
*   устаревших блоков про тарифы, отзывы и абстрактный маркетинг;
*   обещаний, которые не подтверждаются кодовой базой.

## 7. SEO и метаданные

Сайт использует:

*   title и description под реальную модель продукта;
*   Open Graph и Twitter Card;
*   canonical URL;
*   JSON-LD для SoftwareApplication без спекулятивных ценовых обещаний;
*   sitemap через `@astrojs/sitemap`.

## 8. Деплой

Сайт собирается как статический Astro-проект и может размещаться на Apache/Nginx или любом другом хостинге для статических файлов.

Базовый сценарий деплоя:

1.  `cd website`
2.  `npm run build`
3.  содержимое `dist/` публикуется на хостинге

## 9. Критерии качества

Изменение сайта считается успешным, если:

*   главная объясняет продукт простым человеческим языком;
*   верхнее меню короткое и не перегружено;
*   docs не ощущается тупиком;
*   тексты не противоречат текущему приложению;
*   на мобильных и десктопных экранах нет горизонтального переполнения;
*   анимированный фон делает страницу живой, но не мешает чтению.
