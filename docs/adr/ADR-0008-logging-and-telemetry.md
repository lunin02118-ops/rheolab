# ADR-0008: Logging and telemetry — единый фасад логирования

**Статус:** ✅ Реализовано  
**Дата принятия:** 2026-04-17 (WP-5.2)  
**Дата документирования:** 2026-04-18  
**Авторы:** Platform Team  
**Затронутые компоненты:** `src/lib/logger.ts`, `src-tauri/` (tauri-plugin-log)

---

## 1. Контекст

До WP-5.2 в проекте было 3 отдельных логгера (`consoleLogger`, `debugLogger`, ad-hoc `console.log`). Диагностика проблем усложнялась отсутствием единого формата, фильтрации уровней и подписки на лог-поток для UI-отладчика (LogViewer).

## 2. Рассмотренные альтернативы

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **Единый фасад (logger.ts)** | Один API, уровни, подписка, Tauri forwarding | Миграция 200+ вызовов |
| Внешняя библиотека (winston/pino) | Зрелые решения | Серверно-ориентированы, излишни для desktop |
| Sentry/DataDog SDK | Crash reporting, метрики | Требует сеть, неприемлемо для офлайн-приложения |
| Оставить как есть | Нулевые затраты | Неуправляемый шум в консоли |

## 3. Решение

Единый фасад `src/lib/logger.ts`:

- **Уровни:** TRACE → DEBUG → INFO → WARN → ERROR
- **Production:** подавление TRACE/DEBUG
- **Namespace:** `createLogger("module")` — префикс `[MODULE]`
- **Tauri forwarding:** ERROR → `tauri-plugin-log` → файл на диске
- **Подписка:** `subscribeToLogs()` → UI-компонент LogViewer

**Rust-сторона:** Tauri plugin-log для дисковой записи; Rust-код использует стандартный `log` crate.

## 4. Последствия

- **Положительные:** единый grep-friendly формат, управление шумом через уровни, real-time LogViewer для разработчиков
- **Отрицательные:** нет офлайн crash-reporting (осознанный trade-off)
- **Нейтральные:** формат не graylog/ELK-совместим — не требуется для desktop
