# Офлайн-активация корпоративной лицензии

Этот сценарий нужен для рабочих мест без доступа в интернет. Корпоративная
лицензия является постоянной, привязана к железу и активируется через обмен
кодом запроса и ответным кодом активации.

## Где находится генератор

Основной рабочий генератор теперь находится в админке сервера лицензий:

1. Откройте `RheoLab License Admin`.
2. В блоке `Офлайн-активация Corporate` вставьте код запроса клиента.
3. Нажмите `Сформировать офлайн-код`.
4. Скопируйте выданный `RL-ACT1:...` и передайте клиенту.

Админка сама создаёт корпоративный ключ, привязывает его к Machine ID из
запроса и подписывает код активации. Если для этого Machine ID уже есть
активная корпоративная лицензия, она переиспользуется, чтобы повторная выдача
после переустановки ОС не плодила дубликаты.

CLI-генератор оставлен как резервный/операторский инструмент:

```powershell
node scripts\licensing\sign-offline-activation.mjs `
  --request-file request.txt `
  --license-key RHEO-XXXX-XXXX-XXXX `
  --customer "Название компании" `
  --private-key C:\secure\license_private.pem
```

Скрипт принимает код запроса от клиента и выпускает код:

```text
RL-ACT1:...
```

Для корпоративной offline-лицензии срок не задается. `expiresAt` всегда
`null`, а payload подписывается как `type: "corporate"`,
`hardwareBound: true`, `permanent: true`.

## Где это реализовано в программе

Клиентская часть:

- `src/components/licensing/LicenseActivationDialog.tsx` — вкладка
  `Офлайн Corporate`, генерация кода запроса и ввод ответного кода.
- `src/lib/store/license-store.ts` — вызовы Tauri-команд
  `licensing_offline_activation_request` и `licensing_activate_offline`.

Rust/Tauri часть:

- `src-tauri/src/commands/licensing/mod.rs` — IPC-команды для frontend.
- `src-tauri/src/commands/licensing/engine/offline.rs` — генерация request-кода,
  проверка подписи, проверка типа `corporate`, проверка `hardwareBound` и
  сверка Machine ID.
- `src-tauri/src/commands/licensing/hardware/` — расчет Machine ID по железу.
- `src-tauri/src/startup/commands_registry.rs` — регистрация IPC-команд.

## Как активировать рабочее место

1. Запустить RheoLab на целевом компьютере.
2. Открыть окно активации лицензии.
3. Перейти на вкладку `Офлайн Corporate`.
4. Нажать `Сформировать код`.
5. Передать код запроса в поддержку.
6. Поддержка открывает админку сервера лицензий и в блоке
   `Офлайн-активация Corporate` вставляет код запроса.
7. Поддержка возвращает пользователю `RL-ACT1:...`.
8. Пользователь вставляет этот код в поле `Офлайн-ключ активации`.
9. Программа проверяет подпись и привязку к железу локально, без интернета.

Отдельной "корпоративной сборки" сейчас нет: используется обычный desktop build
RheoLab, а корпоративный режим включается активной лицензией типа `corporate`.

## Что будет после переустановки ОС

Локальная запись лицензии хранится в базе приложения и может пропасть при
переустановке ОС. Но Machine ID рассчитывается не из Windows и не из диска, а
из аппаратных идентификаторов:

- CPU `ProcessorId`;
- motherboard UUID;
- BIOS serial.

Поэтому на том же компьютере после переустановки ОС можно снова вставить ранее
выданный `RL-ACT1:...`, если железо не изменилось.

Если ответный activation-код не сохранили, нужно заново сформировать request-код
на переустановленной системе и снова подписать его в поддержке.

Если заменили CPU, материнскую плату или изменились SMBIOS/BIOS serial values,
старый activation-код будет отклонен как выпущенный для другого устройства.
В этом случае нужен новый request-код и новая подпись.

## Как хранить ключи

Есть три разных сущности:

1. `RHEO-XXXX-XXXX-XXXX` — номер лицензии или договора. Хранить в CRM,
   админке сервера лицензирования или внутреннем реестре выдачи.
2. `RL-ACT1:...` — ответный код активации для конкретного
   компьютера. Хранить у клиента и у поддержки в защищенном архиве/менеджере
   секретов вместе с названием компании, датой выдачи и Machine ID. Код нельзя
   использовать на другом железе, но это все равно лицензионный credential.
3. `license_private.pem` — приватный RSA-ключ подписи. Это самый важный секрет.
   Он не должен попадать в репозиторий, клиентскую сборку, почту или обычные
   рабочие папки. Хранить только на сервере лицензирования или защищенной
   support-машине, с правами доступа уровня `600`, плюс encrypted backup.

Код запроса `RL-REQ1:...` постоянный для текущего Machine ID и секретом не
является: он содержит Machine ID и минимальные метаданные запроса, но не
содержит лицензионный ключ и не дает права активировать программу без
серверной подписи.
