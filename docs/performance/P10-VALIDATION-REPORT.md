## Conclusion

**P10 is essentially neutral on these microbench workloads.**

The headline numbers (mean p50 +30%, mean p95 -15%) look dramatic but
they hide what is really happening: **iteration 1 is always 3–5×
slower than iterations 2–5** because the Typst font cache is cold,
and the absolute deltas are 2–4 ms on 10–25 ms baselines — within
run-to-run variance for a single 5-iteration sweep.

Looking at warm samples (iterations 2–5) only:

| Fixture | WITH-P10 mean (warm) | NO-P10 mean (warm) | Δ |
|---|---:|---:|---:|
| n=3   | (9.4 + 11.9 + 13.7 + 13.0) / 4 = **12.0 ms** | (8.9 + 8.8 + 8.9 + 8.6) / 4 = **8.8 ms** | **+3.2 ms (+36%)** |
| n=5   | (10.8 + 10.1 + 14.2 + 16.6) / 4 = **12.9 ms** | (13.5 + 11.4 + 11.5 + 11.4) / 4 = **12.0 ms** | **+0.9 ms (+8%)** |
| n=10  | (23.9 + 26.3 + 19.0 + 18.2) / 4 = **21.9 ms** | (16.9 + 19.8 + 27.1 + 18.8) / 4 = **20.7 ms** | **+1.2 ms (+6%)** |

Warm-only averages: WITH-P10 trails NO-P10 by **+1.7 ms mean / +17 % mean**
on these synthetic fixtures.  That's the opposite of what P10 was hoping for.

**Why P10 did not win here:**

1. **Workload is too small to expose CPU optimisation.** The PDFs are
   54–80 KB.  Most of the time goes into Typst font cache lookups,
   SVG-to-PDF conversion, and small-allocation overhead — none of which
   are in the hot loops `opt-level=3` improves.
2. **`opt-level=3` produces larger code.**  More aggressive inlining /
   unrolling makes the i-cache footprint worse on small workloads.
   `opt-level="s"` (the workspace default) keeps the code dense enough
   to fit in i-cache, which dominates on short benches.
3. **Synthetic fixtures don't exercise the analysis pipeline.** P10 was
   meant to also help analysis (the other reason `rheolab-core` got
   `opt-level=3`), and that path is **not** measured by this microbench
   at all.  The Typst-only measurement is a worst case for P10.

**Binary cost:** `bench_comparison_pdf.exe` size delta is `+1.49 MB`
WITH-P10 (16.98 → 18.47 MB) — proportionally similar to what the full
release binary saw.
