# Audit v2 — Profiling Verification (2026-04-28)

## Why this report exists

Audit v2 raised two performance/storage findings that called for refactoring
work.  Before scheduling that work we measured the **actual impact** of both
items against the live production database and the existing perf baseline
suite.  The numbers turned both findings into **not-actionable in the
current state** — kept here so future audits do not re-open them without
new evidence.

| Audit ID | Finding | Verdict |
|---|---|---|
| **DB-003** | `Experiment.rawPoints TEXT NOT NULL` JSON duplicates the columnar zstd `ExperimentData.dataBlob` | **Not actionable** — rawPoints is empty (`'[]'`, 2 bytes) for 11 669 / 11 669 rows.  Net waste: 23 KB / 400 MB DB ≈ 0.006 %. |
| **PERF-003** | V8 JS heap cap `--max-old-space-size=512` may starve heavy comparison/report workflows | **Not actionable** — measured peak JS heap is **8.4 MB / 512 MB ≈ 1.6 %** in soak, **42.9 MB ≈ 8.4 %** in the 6-fixture workflow.  62× headroom at typical workloads. |

Both findings remain *technically valid* (rawPoints is schema-level
duplication; the JS cap is a hard ceiling) — they are simply ranked
**watch-list, not refactor** in 2026-04-28.

## Methodology

### DB-003 measurement

* **Source:** live production DB at
  `%APPDATA%\com.rheolab.enterprise\rheolab.db` — 400.1 MB on disk,
  11 669 saved experiments, captured 2026-04-28.
* **Mode:** `sqlite3 file:rheolab.db?mode=ro` (read-only URI), no risk
  of contention with the running app.
* **SQL:**
  ```sql
  SELECT COUNT(*) AS n,
         SUM(LENGTH(rawPoints))   AS total_bytes,
         AVG(LENGTH(rawPoints))   AS avg_bytes,
         MAX(LENGTH(rawPoints))   AS max_bytes
  FROM Experiment;

  SELECT COUNT(*) AS n,
         SUM(LENGTH(dataBlob))    AS total_bytes,
         AVG(LENGTH(dataBlob))    AS avg_bytes,
         MAX(LENGTH(dataBlob))    AS max_bytes
  FROM ExperimentData;
  ```

### PERF-003 measurement

* **Source 1 — soak (steady-state / leak detection):**
  `docs/performance/memory-performance-report-2026-04-27.md`,
  20 runs aggregated from
  `outputs/e2e/perf/soak-*.json`.  Generated by
  `npx playwright test --config playwright.tauri-soak.config.ts`.
* **Source 2 — workflow (6 fixtures + comparison + reports):**
  `docs/performance/BASELINES.md` Baseline #1, JSON file
  `outputs/e2e/perf/workflow-1771733913362-tauri.json`.  Generated by
  `npm run perf:workflow:tauri`.
* Both measure `performance.memory.usedJSHeapSize` (V8 old-gen +
  young-gen + code-space, the same pool the `--max-old-space-size`
  cap applies to).

## DB-003 — measured numbers

| | rows | total bytes | avg bytes | max bytes |
|---|---:|---:|---:|---:|
| `Experiment.rawPoints` | 11 669 | **23 338** | 2 | 2 |
| `ExperimentData.dataBlob` | 11 669 | **341 131 004** | 29 233 | 141 635 |

* `rawPoints` averages **2 bytes** per row across 11 669 rows — that is
  the literal string `'[]'` written by the post-migration save path.
* `dataBlob` averages **29 KB** per row — this is the columnar zstd
  payload (`time / shearRate / shearStress / viscosity / temperature`
  arrays) that fully replaces what `rawPoints` used to hold.
* Total `rawPoints` storage: **23 KB** in a **400 MB** DB —
  **0.006 %** of the on-disk footprint.  Removing the column entirely
  would save less storage than a single saved experiment occupies.
* Storage ratio columnar / json ≈ **14 622 ×** in favour of columnar.
  The migration to `ExperimentData.dataBlob` already happened —
  `rawPoints` is a vestigial schema artefact, not a duplication of
  meaningful data.

### Verdict

**Schema cleanup is optional, not urgent.**  A v0008 migration that
converts `rawPoints` to `NULLABLE` (then writes `NULL` instead of
`'[]'`) would remove the technical schema lie, but the on-disk and
read-path savings are well below noise.  Current behaviour is safe;
keep it on the watch-list and revisit if the DB grows past 5 GB.

If the cleanup is scheduled it should be paired with deletion of the
`rawPoints` deserialization shim in the experiment-load path (currently
parses `'[]'` as a no-op fallback).

### Hitting the cap — synthetic estimate

Average per-experiment analysis payload in JS heap (after `decode_blob`
+ derived metrics) ≈ **30 KB** (matches `dataBlob` 29 KB columnar).
To exhaust 512 MB JS old-gen one would need to keep
**≈ 17 000 experiments** simultaneously analysed and resident in JS
without GC pressure.

The comparison view's UI is hard-bounded:

* Comparison flow tops out around **20 experiments** per single canvas
  render (uPlot lane limit + readability cap).
* Reports are generated one-at-a-time and freed.
* Library pagination keeps only the visible page (≤ 50 rows) hydrated.

The realistic upper bound for resident analysis data in current code
is therefore ≈ **20 × 30 KB = 600 KB** — three orders of magnitude
below the cap.

### Verdict

**512 MB cap is not a real ceiling for current workloads.**  Audit's
suggestion to load-test with 1 k / 10 k experiments is fine as a
defence-in-depth exercise, but the existing soak + workflow data
already proves headroom.  No code or config change recommended.

If a future feature unlocks bulk-analysis (e.g. "compare 1000
experiments at once"), the right control is the comparison-flow
hard-cap (UI-side) rather than the JS heap cap (platform-side).
